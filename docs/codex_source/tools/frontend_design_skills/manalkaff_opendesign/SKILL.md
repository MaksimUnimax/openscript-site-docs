## What it says in practice

- read the project and choose a deliberate aesthetic direction before writing UI code
- keep the design anchored to a specific visual system instead of defaulting to generic UI
- write output into the repo-local OpenDesign workspace
- use the design skill as a structured starting point, not as a loose style prompt
- treat the verifier loop as part of the workflow

## Why it may help OpenScript

- can help drive a more deliberate lesson UI than a generic text-heavy page
- encourages a structured design workflow for future lesson pages
- works well as a reference when creating temporary OpenScript UI skills

## Risks

- can be broader than the immediate lesson-rendering problem
- may create more design process than the project needs if used without a narrow brief
